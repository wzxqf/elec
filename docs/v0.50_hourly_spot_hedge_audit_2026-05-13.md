# v0.50 小时级现货修正通道审计与优化路径

## 1. 审计范围

本次审计面向当前 `v0.50` 代码树与正式产物，重点核查小时级现货修正通道为何未在正式滚动结果中形成非零交易量，并提出后续实验路线。

审计对象：

- `experiment_config.yaml`
- `src/training/score_kernel.py`
- `src/backtest/materialize.py`
- `src/analysis/benchmarks.py`
- `outputs/v0.50/raw/metrics/rolling_hourly_results.csv`
- `outputs/v0.50/raw/metrics/rolling_weekly_results.csv`
- `outputs/v0.50/raw/metrics/rolling_parameter_snapshots.csv`
- `outputs/v0.50/raw/metrics/benchmark_comparison.csv`
- `outputs/v0.50/reports/v0.50_human_report.md`

本次只做本地只读复算与静态代码审计，未把结论升格为正式远程验证结果。

## 2. 关键结论

当前小时级现货修正通道在代码中存在，正式产物中没有实际激活。直接原因集中在 `src/training/score_kernel.py` 的小时修正限额基数：

```text
exposure_band_mwh_raw -> raw_spot_hedge_limit_mwh -> raw_spot_hedge_mwh
exposure_band_mwh     -> project_hourly_hedge_tensor() -> spot_hedge_limit_mwh
```

`exposure_band_mwh_raw` 在滚动窗口最优上层粒子下被压到 `0`。随后周度可行域把正式 `exposure_band_mwh` 投影到非零下限，但原始小时修正限额已经用投影前的 `exposure_band_mwh_raw` 计算完成，因此 `raw_spot_hedge_mwh` 全部为 `0`。输出中的 `spot_hedge_limit_mwh` 来自投影后的 `exposure_band_mwh`，所以正式 CSV 会同时出现“限额非零、实际修正全零”的现象。

该问题会让下层粒子在当前正式训练中几乎失去作用。`rolling_parameter_snapshots.csv` 中 10 个滚动窗口的下层参数摘要完全一致，和小时交易量全零一致。

## 4. 代码路径审计

### 4.1 周度风险带宽

`src/training/score_kernel.py` 先生成原始风险带宽：

```python
exposure_band_mwh_raw = relu(
    exposure_band_base_ratio
    * forecast_load
    * (1 + tanh(exposure_band_feature_scale * feature_signal + exposure_bias))
)
```

随后执行周度可行域投影：

```python
exposure_band_mwh = clamp(
    exposure_band_mwh_raw,
    min=exposure_band_ratio_min * forecast_load,
    max=exposure_band_ratio_max * forecast_load,
)
```

当前配置中 `exposure_band_ratio_floor = 0.05`，因此即使 `exposure_band_mwh_raw = 0`，正式 `exposure_band_mwh` 仍会被投影到非零下限。

### 4.2 小时级信号与门控

小时级原始信号使用滞后特征：

```python
price_spread_lag1
load_dev_lag1
renewable_dev_lag1
price_spread_abs_lag1
renewable_dev_abs_lag1
business_hour_flag / peak_hour_flag / valley_hour_flag
```

门控参数来自 `experiment_config.yaml`：

```yaml
score_kernel:
  hourly_gate:
    enabled: true
    signal_deadband: 0.12
    temperature: 0.04
```

只读复算显示，保存的滚动窗口下层参数对应的 `raw_hourly_signal` 并非全部落在死区内。以 10 个滚动测试周计，单周超过 `0.12` 死区的小时数大多在 129 到 154 之间，最后一个不完整周为 84。说明“信号门控过严”不是本轮全零的首要原因。

### 4.3 小时修正限额的断点

当前代码使用投影前带宽计算小时原始限额：

```python
raw_spot_hedge_limit_mwh = (
    exposure_band_mwh_raw.unsqueeze(-1) / valid_hour_count
) * hourly_limit_multiplier
```

当 `exposure_band_mwh_raw = 0` 时，`raw_spot_hedge_limit_mwh = 0`，于是：

```python
raw_spot_hedge_mwh =
    hard_gate * tanh(raw_hourly_signal) * raw_spot_hedge_limit_mwh
```

无论信号是否超过死区，结果都被压到 `0`。

随后 `spot_hedge_limit_mwh` 又使用投影后的 `exposure_band_mwh` 生成：

```python
spot_hedge_limit_mwh =
    exposure_band_mwh / valid_hours * hourly_share_cap
```

这解释了正式产物中“执行限额非零、执行动作全零”的组合。

### 4.4 下层参数退化

当前下层布局为 4 个参数块，每块 12 维：

- `spread_response`
- `load_deviation_response`
- `renewable_response`
- `policy_shrink_response`

但评分核只使用每个块的均值作为单一响应系数。即 48 维下层参数实际压缩为 4 个标量：

```python
spread_coef = mean(spread_response)
load_coef = mean(load_deviation_response)
renewable_coef = mean(renewable_response)
policy_shrink = sigmoid(mean(policy_shrink_response))
```

正式 10 个滚动窗口中，下层参数摘要完全一致：

| 指标 | 数值 |
| --- | ---: |
| `lower_mean_abs` | 0.073065 |
| `spread_mean` | -0.018059 |
| `load_mean` | -0.055588 |
| `renewable_mean` | -0.051380 |
| `shrink_mean` | 0.009267 |

在小时限额已归零的条件下，下层参数对得分没有可见贡献，优化器自然缺少区分信号。

## 5. 优化路径

### 路径 A：修复风险带宽口径

首要实验应把小时原始限额的基数从 `exposure_band_mwh_raw` 改为投影后的 `exposure_band_mwh`。这样小时修正量在合法可行域带宽内生成，后续 `project_hourly_hedge_tensor()` 继续负责份额和爬坡投影。

推荐验收：

- 构造 `exposure_band_mwh_raw = 0`、`exposure_band_mwh` 被 floor 拉起、小时信号超过死区的单元测试。
- 期望 `spot_hedge_mwh` 出现非零值，且不超过 `spot_hedge_limit_mwh`。
- 期望 `friction_cost_w > 0`。

### 路径 B：小时信号标准化

当前 `price_spread_lag1`、`load_dev_lag1`、`renewable_dev_lag1` 以原始单位进入信号，权重和死区的含义容易随数据量纲变化。建议新增只用历史信息计算的稳健标准化：

- 价差信号除以前一滚动训练段或前一周的 `price_spread` MAD/IQR。
- 负荷偏差信号除以历史 `net_load_da` 或 `load_dev` 稳健尺度。
- 新能源偏差信号除以历史 `renewable_dev` 稳健尺度。
- 标准化尺度进入 `feature_manifest` 和正式报告，避免论文中无法解释死区含义。

### 路径 C：门控死区实验

在标准化前，`signal_deadband = 0.12` 是原始量纲阈值；标准化后可作为无量纲阈值解释。建议实验网格：

| 参数 | 候选值 |
| --- | --- |
| `signal_deadband` | 0.00, 0.05, 0.10, 0.15, 0.20 |
| `temperature` | 0.03, 0.05, 0.08 |

筛选指标：

- `nonzero_hour_share`
- `spot_abs_sum_mwh`
- `friction_cost_w`
- `excess_profit_w`
- `cvar99_w`
- `hedge_error_w`

### 路径 D：风险带宽与政策收缩实验

现有可行域在研究后期持续触发：

- `exposure_band_ratio_max = 0.12`
- `max_hourly_hedge_share = 0.4`
- `max_hourly_ramp_share = 0.2`

建议先固定政策可行域，只调整评分核生成带宽：

| 参数 | 候选值 |
| --- | --- |
| `exposure_band_base_ratio` | 0.05, 0.10, 0.15, 0.20 |
| `exposure_band_feature_scale` | 0.05, 0.10, 0.15 |
| `hourly_limit.base_multiplier` | 0.25, 0.50, 0.75, 1.00 |
| `hourly_limit.shrink_multiplier` | 0.00, 0.25, 0.50 |

约束口径保持不变，便于论文中说明实验是在制度可行域内寻找更合理的下层激活强度。

### 路径 E：交易摩擦权重实验

当前摩擦成本口径为：

```text
friction_cost_w = sum(abs(spot_hedge_mwh)) * 1.20
reward -= lambda_trade * friction_cost_w
```

正式结果中 `friction_cost_w = 0`，该项尚未发挥作用。激活现货修正后应同步扫描：

| 参数 | 候选值 |
| --- | --- |
| `friction_cost_yuan_per_mwh` | 0.60, 1.20, 2.40, 4.80 |
| `reward.lambda_trade` | 0.05, 0.10, 0.20, 0.40 |

验收条件应采用“利润改善 + 尾部风险不过度恶化”的双约束，例如：

```text
sum(excess_profit_w) > baseline_run_sum_excess_profit_w
mean(cvar99_w) <= baseline_run_mean_cvar99_w * 1.03
mean(hedge_error_w) <= baseline_run_mean_hedge_error_w
```

### 路径 F：下层参数搜索重构

当前 48 维下层参数实际压缩成 4 个均值，搜索空间存在冗余。建议两阶段处理：

1. 短期：保留 48 维配置，新增诊断列记录各块均值、信号开门小时数和非零修正小时数。
2. 中期：把下层布局改成显式小维度参数，例如每个信号使用 `coef, asymmetry, session_gain, smooth_penalty`，让 48 维真正参与分时响应。

该路径需要单元测试覆盖维度编译、旧模型兼容和物化结果字段。

## 6. 实验分层建议

建议按三层推进，避免一次性修改过多因素。

### 第一层：断点修复实验

只改小时限额基数：

```text
raw_spot_hedge_limit_base = exposure_band_mwh
```

保持信号、门控、摩擦权重和下层搜索参数不变。目标是证明非零小时修正能够从现有信号自然产生。

### 第二层：参数敏感性实验

在第一层通过后，做局部网格：

- `signal_deadband`
- `hourly_limit.base_multiplier`
- `friction_cost_yuan_per_mwh`
- `lambda_trade`

sum_profit_baseline_w

## 8. 审计结论

小时级现货修正没有激活的首要技术断点已经明确：小时原始交易限额使用投影前风险带宽，且正式最优上层粒子把该原始带宽压到 0。后续优化应先修复这一口径，再讨论门控死区、信号标准化、交易摩擦权重和下层参数搜索。

断点修复已经进入代码，既有 `outputs/v0.50/` 正式数值仍来自修复前运行。后续需要通过远程 Jupyter 全流程刷新产物，才能判断非零现货修正是否在不显著恶化尾部风险的情况下改善强基准比较。
