# v0.33 参数化双层混合粒子群主线切换设计

## 背景

当前仓库虽然已将 `v0.33` 文档表述为“双层混合粒子群主线”，但正式代码路径仍混杂旧口径：

- 一部分模块已切换到 `hybrid_pso + tensor score kernel`
- 一部分回测、输出字段和约束逻辑仍沿用旧的 `lock_ratio / delta_lock / clip` 语义
- 收益函数、奖励函数、尾部风险统计口径尚未在文档与代码中统一

本设计将项目主线正式切换为论文新的程序主模型：**参数化双层混合粒子群模型**。该切换是断代式替换，不保留旧字段作为正式口径，不继续兼容旧业务语义。

## 目标

本次主线切换完成后，项目应满足以下目标：

1. 论文主模型正式改为参数化双层混合粒子群模型。
2. 上层不再输出二维动作，而是直接输出周度参数化持仓策略。
3. 下层不再是独立训练 DRL，而是小时级滚动修正规则粒子层。
4. 合约调整量仅受政策规则投影约束，不再施加额外极端裁剪。
5. 经济收益函数与训练奖励函数分离定义。
6. 尾部风险正式统一为 `CVaR_99%`。
7. 正式文档、正式配置、正式代码和正式输出字段全部改用新口径。

## 非目标

本次切换不包含以下内容：

- 恢复 PPO / SB3 / gym 训练栈
- 引入与论文无关的新市场机制或新主体
- 维持旧 `lock_ratio` 主语义的兼容输出
- 将 15 分钟层升级为单独训练层

## 正式模型定义

### 1. 总体结构

正式主链定义为：

`data -> policy_deep -> tensor_bundle -> parametric_hybrid_pso -> policy_projection -> rolling_retrain -> 15m_settlement -> reporting`

其中：

- `policy_deep` 负责政策文本清单、结构化规则和制度状态轨迹
- `tensor_bundle` 负责周度、小时级和 15 分钟结算相关张量编译
- `parametric_hybrid_pso` 为双层混合粒子群主训练器
- `policy_projection` 为唯一正式业务约束器
- `rolling_retrain` 为唯一正式主回测调度方式
- `15m_settlement` 为唯一正式结算评估口径

### 2. 上层参数化策略

上层粒子 `θ^U` 表示周度参数化持仓策略，而不是二维动作。

对于每个周 `w`，上层策略输出：

- `contract_adjustment_mwh_raw_w`
- `contract_adjustment_mwh_exec_w`
- `contract_position_mwh_w`
- `exposure_band_mwh_w`
- `contract_curve_h1_w` 至 `contract_curve_h24_w`
- `policy_gate_w`

含义如下：

- `contract_adjustment_mwh_raw_w`：策略原始建议的周度合约调整量
- `contract_adjustment_mwh_exec_w`：经政策投影后的可执行周度合约调整量
- `contract_position_mwh_w`：本周最终合约持仓量
- `exposure_band_mwh_w`：小时级现货边际风险带宽
- `contract_curve_h1_w` 至 `contract_curve_h24_w`：24 小时中长期合约分配曲线
- `policy_gate_w`：用于表达制度门控、制度收缩和政策状态下可行域变化的参数

### 3. 下层小时级滚动修正规则

下层粒子 `θ^L` 表示小时级滚动修正规则参数。

对于每个周 `w`、小时 `h`，下层输出：

- `spot_hedge_mwh_{w,h}`
- `spot_hedge_limit_mwh_{w,h}`
- `spread_response_{w,h}`
- `load_deviation_response_{w,h}`
- `renewable_disturbance_response_{w,h}`
- `policy_shrink_multiplier_{w,h}`

下层目标是在上层给定的周度合约持仓与现货带宽内，依据价差、负荷偏差、新能源扰动和制度收缩状态，生成小时级现货修正量。

### 4. 约束与投影

本设计明确规定：**合约调整量只受政策规则投影约束，不作其他极端裁剪。**

这意味着：

- 不再使用 `lock_ratio_min / lock_ratio_max / delta_lock_cap`
- 不再使用多次 `clip`、`soft_clip`、`hard_clip` 作为正式业务约束链
- 不再用经验上限、经验下限替代政策约束

唯一允许保留的保护逻辑是：

- 数值稳定处理，如防止 `nan`、除零、空张量异常
- 由制度规则明确给出的可行域投影

投影流程必须为：

`原始参数 -> 参数化策略输出 -> 政策规则投影 -> 可执行电量决策`

若出现超出政策可行域的原始输出，应执行：

1. 政策投影生成可执行值
2. 记录 `policy_binding_flags`
3. 记录 `policy_violation_penalty_w`

不得再额外叠加经验性极端裁剪。

## 正式经济收益函数

### 1. 周度经济收益

正式经济收益按周定义为：

`Π_w = R_retail,w - C_lt,w - C_spot,w - C_imb,w - C_adj,w - C_risk,w`

其中：

- `R_retail,w`：零售侧售电收入
- `C_lt,w`：中长期采购成本
- `C_spot,w`：日前 / 日内现货采购成本
- `C_imb,w`：15 分钟代理结算下的不平衡成本
- `C_adj,w`：合约调整与现货修正带来的调整成本
- `C_risk,w`：尾部风险对应的资本占用或风险映射成本

总样本经济收益定义为：

`Π = Σ_w Π_w`

### 2. 尾部风险统计

正式尾部风险统计口径统一为：

- 基于 15 分钟代理结算损益或采购成本序列
- 采用 `CVaR_99%`
- 即关注最差 `1%` 尾部样本的平均损失

项目中所有未特别说明的 `CVaR`，均默认指 `CVaR_99%`。

## 正式训练奖励函数

训练优化目标不直接等于 `Π`，而是最大化相对强基准的风险调整后收益。

周度奖励函数定义为：

`J_w = G_w - λ_tail * CVaR99_w - λ_hedge * HedgeErr_w - λ_trade * Fric_w - λ_violate * Viol_w`

其中：

- `G_w = Π_w - Π_w^baseline`
- `baseline` 固定为 `dynamic_lock_only`
- `CVaR99_w`：基于 15 分钟结算序列的 `CVaR_99%`
- `HedgeErr_w`：套保误差
- `Fric_w`：交易摩擦与调节平滑成本
- `Viol_w`：政策规则违反或政策绑定惩罚

正式约束如下：

- `profit` 字段只表示经济收益结果
- `reward` 字段只表示训练目标
- 不再使用 `reward = -procurement_cost`
- 不再把采购成本直接当作奖励函数的全部定义

## 配置结构重构

`experiment_config.yaml` 应围绕新主线重构为以下核心章节：

- `project`
- `data`
- `policy_regime`
- `policy_projection`
- `upper_strategy`
- `lower_strategy`
- `hybrid_pso`
- `economics`
- `reward`
- `rolling_retrain`
- `reporting`

其中：

### `policy_projection`

- 定义政策投影规则
- 定义政策绑定时的投影与惩罚方式
- 禁止配置额外经验裁剪

### `upper_strategy`

- 定义上层参数布局
- 定义合约调整、带宽、24 小时曲线和政策门控参数

### `lower_strategy`

- 定义小时级价差、负荷偏差、新能源扰动、平滑和制度收缩参数

### `economics`

- 定义零售收入口径
- 定义采购成本口径
- 定义偏差结算口径
- 定义摩擦成本口径
- 定义尾部风险映射口径

### `reward`

- `baseline_strategy = dynamic_lock_only`
- `cvar_alpha = 0.99`
- `lambda_tail`
- `lambda_hedge`
- `lambda_trade`
- `lambda_violate`

## 代码改造范围

本次切换应按以下顺序整体替换：

1. 入口与配置：`run_all.py`、`src/scripts/run_pipeline.py`、`src/scripts/train.py`、`src/scripts/evaluate.py`、`src/scripts/backtest.py`、`src/config/load_config.py`、`experiment_config.yaml`
2. 粒子解释与策略输出：`src/agents/hybrid_pso.py`，必要时新增参数编码和政策投影模块
3. 训练热路径：`src/training/tensor_bundle.py`、`src/training/score_kernel.py`
4. 评估与结算：`src/backtest/materialize.py`、`src/backtest/settlement.py`、`src/backtest/metrics.py`、`src/backtest/rolling_pipeline.py`
5. 旧语义清理：`src/backtest/simulator.py`、`src/backtest/benchmarks.py` 及所有旧字段引用点
6. 测试：配置、入口、参数化输出、政策投影、`CVaR_99%`、收益奖励一致性、回测产物

## 验收标准

本设计完成后，应满足以下验收标准：

1. 正式文档中不再将 `lock_ratio` 作为主决策变量。
2. 正式代码路径中不再使用旧字段或旧裁剪链。
3. `CVaR_99%` 在配置、计算、报告和测试中完全一致。
4. 合约调整量只受政策规则投影约束，不存在额外极端裁剪。
5. `profit_w` 与 `reward_w` 同时输出，且含义严格区分。
6. `run_all.py` 生成的正式产物只包含新口径字段。
7. 文档、配置、代码、输出字段和测试断言之间不存在旧语义残留。

## 风险与控制

本次切换的主要风险不是单个实现错误，而是旧语义残留。应重点防止以下情况：

- 文档已切换，新代码仍引用旧字段
- 训练评分核已切换，回测物化仍输出旧逻辑字段
- 配置已切换，测试仍断言旧 `lock_ratio` 口径

因此，本次切换必须以“先删除旧主语义，再写入新主语义”为实施原则，不接受文档层面切换但代码层面保留业务兼容。
