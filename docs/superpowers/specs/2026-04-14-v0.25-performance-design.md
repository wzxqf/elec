# v0.25 Performance Design

## Context

`v0.25` is a performance optimization release for the formal training phase. The release must preserve strategy semantics, market interpretation, reward behavior, settlement behavior, and output meaning. The optimization target is execution efficiency only.

The current implementation spends a meaningful share of wall-clock time in environment-side weekly simulation rather than only in PPO forward/backward passes. The main repeated costs are:

- repeated weekly slicing from `bundle["quarter"]`, `bundle["hourly"]`, and `bundle["weekly_metadata"]` inside `simulate_week()`
- repeated `set_index(...).loc[...]` lookups for weekly feature rows and dynamic benchmark rows
- repeated per-call DataFrame filtering and copies in training, evaluation, and backtest loops
- limited device handling that supports `cpu` and `cuda` reporting paths but does not provide a clean optional `mps` path for Apple Silicon

## Goals

1. Reduce environment-side repeated data access overhead during training, evaluation, and backtest.
2. Preserve all business semantics and policy-timing semantics.
3. Keep default training behavior conservative by leaving the default device on `cpu`.
4. Add an optional device-resolution path so Apple Silicon can use `mps` when explicitly requested or when `auto` is chosen later.
5. Update `v0.25.md` and `CHANGELOG.md` so the release scope is explicit and reproducible.

## Non-Goals

- No changes to PPO algorithm choice, action semantics, reward structure, settlement logic, or policy interpretation.
- No additional parallel environments or environment architecture changes.
- No refactor that changes exported result columns or experiment output structure.
- No forced switch of the default training device from `cpu` to `auto` or `mps`.

## Approach Options

### Option A: Weekly slice cache only

Pre-slice quarter/hourly/weekly metadata by `week_start` and use the cache in `simulate_week()`.

Pros:

- lowest implementation risk
- directly addresses the hotspot named in `v0.25.md`

Cons:

- leaves adjacent repeated indexing hotspots in place
- likely under-delivers on total speedup

### Option B: Balanced engineering optimization

Add weekly slice caches, add prebuilt weekly lookup indices, reduce repeated DataFrame filtering/index creation, and add optional device resolution for `auto|cpu|mps|cuda` while keeping default config on `cpu`.

Pros:

- best expected speedup-to-risk ratio
- remains purely semantic-preserving
- removes multiple repeated lookup costs on the same hot path
- gives a clean Apple Silicon path without forcing it as default

Cons:

- broader surface area than a single cache
- requires focused regression testing

### Option C: Aggressive ndarray/vectorization rewrite

Push more simulation logic out of pandas into lower-level arrays and attempt broader device-side acceleration.

Pros:

- highest theoretical upside

Cons:

- highest regression risk
- exceeds the intended scope of a performance-only release

## Decision

Adopt **Option B**.

This release will treat performance as an engineering-path optimization problem rather than a modeling change. The major improvements will remain inside data access and device-selection plumbing while preserving all existing strategy semantics.

## Detailed Design

### 1. Weekly bundle runtime cache

Add a bundle preparation step that builds reusable weekly lookup structures after bundle assembly and after policy-state merges are complete.

Expected cache content:

- `bundle["quarter_by_week"]`: `dict[pd.Timestamp, pd.DataFrame]`
- `bundle["hourly_by_week"]`: `dict[pd.Timestamp, pd.DataFrame]`
- `bundle["weekly_metadata_by_week"]`: `dict[pd.Timestamp, pd.Series]`
- `bundle["weekly_feature_by_week"]`: `dict[pd.Timestamp, pd.Series]`
- `bundle["reward_reference_by_week"]`: `dict[pd.Timestamp, pd.Series]`

Design rules:

- cache keys use normalized `pd.Timestamp`
- cached frames are built once per prepared bundle
- read paths should use the cache directly and avoid repeated `loc[frame["week_start"] == week]`
- no business fields are added or removed from the exposed weekly results

### 2. Simulator lookup optimization

Update the weekly read path in `src/backtest/simulator.py`.

Changes:

- replace `_get_week_frames()` repeated filtering with direct cache reads when caches are present
- fall back to the current behavior if a cache is absent, so the function remains robust
- replace repeated reward-reference filtering with `reward_reference_by_week`
- replace repeated dynamic-base weekly lookup with `weekly_feature_by_week`

This keeps runtime compatibility while ensuring prepared project contexts use the faster path.

### 3. Benchmark lookup optimization

Update `src/backtest/benchmarks.py` so dynamic benchmark base reads can accept prebuilt weekly rows or use cached lookup tables rather than calling `set_index("week_start")` inside every weekly access.

Target effect:

- `get_dynamic_lock_base_for_week()` becomes O(1) lookup on prepared bundles
- benchmark action construction also avoids rebuilding an index for every call path

### 4. Environment observation lookup optimization

`ElecEnv` already builds `feature_frame = weekly_features.set_index("week_start").sort_index()`, which is acceptable because it happens once per environment instance. Keep this pattern unless a shared prepared lookup can be reused cleanly without making the env code harder to maintain.

The main target remains the repeated simulation-side lookups, not the one-time env initialization.

### 5. Device resolution for CPU, CUDA, and MPS

Add a small device-resolution helper in the training path.

Supported config values:

- `cpu`
- `cuda`
- `mps`
- `auto`

Resolution rules:

- `cpu`: always use CPU
- `cuda`: use CUDA if available, otherwise fail fast with a clear error
- `mps`: use MPS if available, otherwise fail fast with a clear error
- `auto`: prefer `cuda`, then `mps`, then `cpu`

Release policy for `v0.25`:

- keep [experiment_config.yaml](/Users/dk/py/elec/experiment_config.yaml:1) default on `cpu`
- document `auto` and `mps` as optional acceleration paths
- update summary logic so GPU reporting treats both `cuda` and `mps` as GPU-backed devices

### 6. Documentation updates

Overwrite `v0.25.md` so it reflects the approved scope:

- weekly pre-slice cache
- adjacent lookup/index optimization on the same hot path
- optional `mps` support without changing the default device

Update `CHANGELOG.md` with a concise `v0.25` entry that explains the same boundary.

## Data Flow

1. `prepare_project_context()` builds the base bundle.
2. Policy traces are merged into weekly metadata and weekly features.
3. A new cache-preparation step adds week-based lookup maps to the bundle.
4. Training/evaluation/backtest continue to call existing public functions.
5. `simulate_week()` uses cache-first reads and falls back only when necessary.
6. PPO device selection resolves from config through the new helper before model creation.

## Error Handling

- Missing cache entries should raise explicit week-level lookup errors rather than silently returning empty frames.
- Unsupported device strings should raise a configuration error.
- Forced `cuda` or forced `mps` should report unavailability clearly.
- `auto` must never fail just because an accelerator is unavailable; it should fall back deterministically.

## Testing Plan

### Unit tests

- cache builder returns complete week-indexed mappings for a small synthetic bundle
- cache-backed `_get_week_frames()` returns the same content as the legacy filter path
- dynamic-base weekly lookup returns the same value before and after optimization
- device-resolution helper returns expected resolved device for mocked availability states
- GPU reporting treats `mps` as GPU-backed

### Regression checks

- existing `tests/test_run_all_script.py` remains green
- at least one representative prepared context can run through training/backtest entry points without schema regressions
- weekly summary outputs remain column-compatible

## Risks and Mitigations

### Risk: cached mutable frames are accidentally mutated across call sites

Mitigation:

- keep explicit copy boundaries where downstream code mutates a frame
- cache the source slice once, then copy only when a consumer needs writable working data

### Risk: broader optimization unintentionally changes semantics

Mitigation:

- preserve current public interfaces
- compare legacy and cache-backed weekly retrieval in tests
- do not alter any formula, reward component, or settlement field

### Risk: MPS is slower than CPU for this workload

Mitigation:

- keep `cpu` as default
- expose `mps` and `auto` as optional paths only
- document that the expected main gain comes from environment-side optimization, not from device switching
