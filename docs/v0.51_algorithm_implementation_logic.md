## 2. 总体算法链路

当前主链可以概括为：

```text
experiment_config.yaml
  -> prepare_project_context
  -> 数据清洗与周度/小时/15分钟样本构造
  -> 政策文件解析与制度状态轨迹
  -> policy_feasible_domain 可行域编译
  -> parameter_compiler 参数块布局
  -> TrainingTensorBundle 张量打包
  -> HYBRID_PSO_V040 双层粒子群训练
  -> batch_score_particles 评分核
  -> materialize_particle_pair 物化周度、小时级、15分钟结果
  -> evaluate / rolling backtest / benchmarks / ablations / robustness
  -> reports、metrics、models、manifest
```

正式运行时，`src/scripts/run_pipeline.py` 依次调用：

1. `prepare_project_context()`：生成数据、政策状态、可行域、参数布局、张量包和基础审计产物。
2. `run_train()`：在训练集上运行 `HYBRID_PSO_V040`，保存 `hybrid_pso_model.json` 和训练轨迹。
3. `run_evaluate()`：在验证集上物化最优粒子，生成验证指标与政策风险调整指标。
4. `run_backtest()`：按滚动重训窗口重新训练模型，生成滚动回测、基准比较、消融和稳健性产物。
5. `_build_human_report()` 与 `_build_ai_structured_report()`：在全流程数值结果生成后一次性汇总，`reports/` 只保留人工报告和 AI 结构化报告。

远程正式验证由根目录 `run_remote_jupyter.py` 触发，上传当前工作区到 Jupyter，在远程 `torch311` 环境中执行 `python run_all.py` 和 `python -m src.scripts.run_pytest tests -q`，然后把产物拉回 `outputs/v0.51/raw/remote_jupyter/<run_id>/`。

## 3. 数据层实现

### 3.1 原始数据读取与清洗

数据入口在 `src/scripts/common.py`：

- `locate_total_csv()` 按 `data.data_candidates` 定位 `total.csv`。
- `load_raw_total_csv()` 读取原始表。
- `clean_total_data()` 按 `sample_start`、`sample_end`、`expected_frequency_minutes=15` 做样本区间过滤、重复值处理、频率检查和质量报告。

当前样本范围：

- 建模样本：`2025-11-01 00:00:00` 至 `2026-03-20 23:45:00`
- 扫描缓冲：`2026-03-28 23:45:00`
- 15 分钟结算间隔：`settlement_interval_hours = 0.25`

### 3.2 派生特征

`src/data/feature_engineering.py` 在 15 分钟层生成基础派生量：

- `net_load_da = 省调负荷_日前 - 新能源负荷-总加_日前`
- `net_load_id = 省调负荷_日内 - 新能源负荷-总加_日内`
- `price_spread = 全网统一出清价格_日内 - 全网统一出清价格_日前`
- `load_dev = net_load_id - net_load_da`
- `wind_dev = 新能源负荷-风电_日内 - 新能源负荷-风电_日前`
- `solar_dev = 新能源负荷-光伏_日内 - 新能源负荷-光伏_日前`
- `renewable_dev = wind_dev + solar_dev`
- `business_hour_flag`、`peak_hour_flag`、`valley_hour_flag`
- `net_load_da_mwh = net_load_da * 0.25`
- `net_load_id_mwh = net_load_id * 0.25`

随后 `aggregate_hourly()` 把 15 分钟数据按小时均值聚合，并按周生成 `hour_index_in_week`，同时为价差、负荷偏差、新能源偏差等生成一阶滞后特征。

### 3.3 周度特征

`build_weekly_features()` 使用前一周小时数据构造当前周状态，避免直接用当前周未来信息。主要统计项包括：

- 日前、日内价格的均值、标准差、最大值、最小值和分位数。
- 价差、绝对价差、负荷偏差、新能源偏差、可再生比例的统计量。
- 工作时段、峰时段、谷时段价差信号。
- 联络线、水电、非市场化机组等扰动代理。
- `weekofyear_sin`、`weekofyear_cos`。

`src/data/weekly_builder.py` 同时构造 `weekly_metadata`，其中包括：

- `forecast_weekly_net_demand_mwh`：按小时日前净负荷求和。
- `actual_weekly_net_demand_mwh`：按小时日内净负荷求和。
- `da_price_mean`、`id_price_mean`、`spread_mean`。
- `lt_price_w_prev_week_da_proxy`：保留上一周日前均价代理，作为审计追踪字段。
- `lt_price_w_effective` 与 `lt_price_w`：评分和结算统一使用的中长期 effective price。2026-02 前取上一周日前均价代理，2026-02 起按 40% 日前均价和 60% 日内均价生成。
- `lt_price_source`、`lt_price_fixed_ratio`、`lt_price_linked_ratio`：记录 2026-02 前后价格口径标签与比例。

当前 `weekly_metadata.csv` 中，2026-02 起 `lt_price_source` 为 `linked_mix_40da_60id`，比例记录为 `0.4/0.6`，`lt_price_w_effective` 进入 `TrainingTensorBundle.lt_weekly_price`。

## 4. 政策状态与可行域

### 4.1 政策文件进入建模流程

政策目录为 `政策环境/`。当前正式产物显示：

- `policy_file_inventory.csv`：18 个政策文件。
- `policy_parse_failures.csv`：0 条解析失败。
- `policy_rule_table.csv`：36 条结构化规则。

结构化规则表字段包括：

```text
rule_id, policy_name, publish_time, effective_start, effective_end,
rule_type, scope, state_group, state_name, state_value, source_file, note
```

政策链条由 `src/policy_deep/regime_builder.py` 与 `src/policy/policy_regime.py` 生成周度状态。当前状态结构保留：

- 当前制度状态，例如 `renewable_mechanism_active`、`lt_price_linked_active`、`ancillary_freq_reserve_tight`。
- 四组前瞻制度状态，例如 `forward_price_linkage_days`、`forward_mechanism_execution_days`、`forward_ancillary_coupling_days`、`forward_info_forecast_boundary_days`。
- 文件统计状态，例如 `policy_count`、`policy_source_file_count`、`policy_parse_failure_count`。

这些状态进入 `weekly_metadata`、`weekly_features`、`policy_state_trace`，其中数值型状态还会进入 `policy_tensor`。

### 4.2 可行域编译

`src/policy/feasible_domain.py` 将制度状态编译为周度与小时级边界。

周度边界包括：

- `contract_adjustment_ratio_min`
- `contract_adjustment_ratio_max`
- `exposure_band_ratio_min`
- `exposure_band_ratio_max`
- `max_hourly_hedge_share`
- `max_hourly_ramp_share`
- `non_negative_position_required`
- `feasible_domain_triggered`

核心规则：

- 若 `lt_price_linked_active` 生效，周度合约调整比例上限从默认 `0.30` 收紧到 `0.12`。
- 若 `ancillary_freq_reserve_tight` 生效，现货敞口带宽上限从默认 `0.25` 收紧到 `0.12`。
- 若 `ancillary_freq_reserve_tight` 生效，小时级对冲份额上限从 `1.00` 收紧到 `0.40`。
- 若 `renewable_mechanism_active` 生效，小时级爬坡份额上限从 `1.00` 收紧到 `0.20`。
- `settlement_semantics` 记录 `previous_week_da_proxy` 或 `linked_40_60`。

`src/policy/projection.py` 提供两类投影：

- `project_weekly_actions()`：把周度合约调整与敞口带宽裁剪到可行域范围，并返回投影缺口和原因。
- `project_hourly_hedge_tensor()`：按小时级单点份额上限和相邻小时爬坡上限投影现货修正量。

## 7. 双层 HPSO 训练

`src/agents/hybrid_pso.py` 是 `HYBRID_PSO_V040` 的主体实现。

当前配置：

```text
upper.particles: 192
lower.particles: 160
iterations: 320
init_scale: 0.12
inertia: 0.68
cognitive: 1.30
social: 1.45
position_clip_abs: 1.25
seed: 42
device: cuda, allow_cpu: true
```

训练流程：

1. 根据 `compiled_parameter_layout` 获取真实上下层维度。
2. 用 `init_scale * N(0, 1)` 初始化上层粒子矩阵和下层粒子矩阵。
3. 初始化上下层速度、个体最优和全局最优。
4. 每轮调用 `batch_score_particles()`，对所有上层粒子和下层粒子组合做批量评分。
5. 对每个上层粒子取所有下层组合中的最小目标值，更新上层个体最优。
6. 对每个下层粒子取所有上层组合中的最小目标值，更新下层个体最优。
7. 在完整二维组合矩阵中寻找全局最优上层粒子和下层粒子。
8. 按标准 PSO 公式更新速度与位置：

```text
v = inertia * v
  + cognitive * rand1 * (personal_best - position)
  + social * rand2 * (global_best - position)

position = clip(position + v, -position_clip_abs, position_clip_abs)
```

9. 保存全局最优 `upper_best`、`lower_best` 和 `best_score`。
10. 写出 `metrics/hybrid_pso_training_trace.csv`。

目标值定义为 `total_score = -sum(reward_w)`，因此粒子群在代码中执行最小化，经济含义等价于最大化累计奖励。

## 8. 评分核

`src/training/score_kernel.py` 的 `batch_score_particles()` 是当前最关键的算法函数。它把粒子参数、周度状态、小时状态、15 分钟价格、可行域边界联结起来，一次性产出收益、风险、约束和奖励。

### 8.1 上层周度信号

启用编译布局时，评分核按参数块切片：

```text
feature_signal_w = weekly_feature_matrix_w @ weekly_feature_weights
policy_signal_w  = policy_feature_matrix_w  @ policy_feature_weights
```

其中：

- `weekly_feature_matrix` 来自 148 维 agent 特征。
- `policy_feature_matrix` 来自配置指定的 6 个制度状态。
- `action_head` 的前两个元素分别作为合约调整和敞口带宽的偏置。

### 8.2 周度合约调整

原始合约调整量：

```text
contract_adjustment_mwh_raw
  = contract_adjustment_scale_ratio
    * tanh(
        contract_adjustment_feature_scale * feature_signal
      + contract_adjustment_policy_scale  * policy_signal
      + contract_bias
    )
    * forecast_weekly_load
```

当前默认参数：

- `contract_adjustment_scale_ratio = 0.30`
- `contract_adjustment_feature_scale = 0.15`
- `contract_adjustment_policy_scale = 0.05`

执行合约调整量：

```text
contract_adjustment_mwh_exec
  = clamp(
      contract_adjustment_mwh_raw,
      contract_adjustment_ratio_min * forecast_weekly_load,
      contract_adjustment_ratio_max * forecast_weekly_load
    )
```

最终中长期底仓：

```text
contract_position_raw
  = contract_position_base_ratio * forecast_weekly_load
  + contract_adjustment_mwh_exec

contract_position_mwh
  = max(contract_position_raw, 0)     # 当 non_negative_position_required 生效
```

当前 `contract_position_base_ratio = 0.60`。

物化结果中保留：

- `lock_ratio_base`
- `delta_lock_ratio`
- `lock_ratio_final`
- `contract_adjustment_mwh_raw`
- `contract_adjustment_mwh_exec`
- `contract_position_mwh`

### 8.3 现货敞口带宽

原始敞口带宽：

```text
exposure_band_mwh_raw
  = relu(
      exposure_band_base_ratio
      * forecast_weekly_load
      * (1 + tanh(exposure_band_feature_scale * feature_signal + exposure_bias))
    )
```

当前默认参数：

- `exposure_band_base_ratio = 0.20`
- `exposure_band_feature_scale = 0.10`

投影后带宽：

```text
exposure_band_mwh
  = clamp(
      exposure_band_mwh_raw,
      exposure_band_ratio_min * forecast_weekly_load,
      exposure_band_ratio_max * forecast_weekly_load
    )
```

该带宽控制小时级现货净修正的总空间。

### 8.4 24 小时合约曲线

上层 `contract_curve_latent` 经过 `_curve_basis()` 形成 24 小时曲线 logits：

```text
contract_curve_logits = contract_curve_latent @ basis_24h
contract_curve        = softmax(contract_curve_logits)
```

评分核把周度中长期底仓按该曲线分配到每个小时：

```text
contract_hour_weight_h
  = contract_curve_h / sum(valid_contract_curve_h)

contract_hourly_h
  = contract_position_mwh * contract_hour_weight_h
```

物化层额外计算：

- `contract_curve_h1` 到 `contract_curve_h24`
- `curve_match_score_w`
- `stability_score_w`

这些字段用于解释合约曲线与负荷形态的贴合度。

### 8.5 下层小时级现货修正

下层参数块在评分核中被压缩为响应系数：

- `spread_coef`
- `load_coef`
- `renewable_coef`
- `policy_shrink`

小时级基础信号：

```text
raw_hourly_signal
  = spread_weight    * spread_coef    * price_spread
  + load_dev_weight  * load_coef      * load_dev
  + renewable_weight * renewable_coef * renewable_dev
```

当前实现允许比较不同变量构造。`hourly_signal.transform = raw` 时直接使用原始价差、负荷偏差和新能源偏差；`forecast_load_normalized` 候选会先对价差和电量偏差做尺度化处理：

```text
price_spread_norm   = price_spread / price_spread_scale_yuan_per_mwh
spread_abs_norm     = abs(price_spread) / price_spread_scale_yuan_per_mwh
load_dev_norm       = load_dev / max(forecast_load_mwh_h, min_load_scale_mwh)
renewable_dev_norm  = renewable_dev / max(forecast_load_mwh_h, min_load_scale_mwh)
renewable_abs_norm  = abs(renewable_dev) / max(forecast_load_mwh_h, min_load_scale_mwh)
```

本轮只读扫描表明，按预测负荷归一化后的变量构造没有改善强基准族超额收益，因此当前正式入口仍采用 `raw` 构造。扫描候选曾尝试用 `signal_clip_abs = 0.75` 抑制极端小时信号，完整远程重训后没有改善滚动收益，当前正式入口保留 `signal_clip_abs = 0.0`，即不启用信号裁剪：

```text
hourly_signal = raw_hourly_signal
```

附加信号：

```text
+ spread_abs_weight
   * spread_coef
   * abs(price_spread)
   * session_signal

+ renewable_abs_weight
   * renewable_coef
   * abs(renewable_dev)
   * (renewable_valley_mix * valley_hour_flag
      + renewable_business_mix * business_hour_flag)
```

时段权重：

```text
session_signal
  = business_hour_weight * business_hour_flag
  + peak_hour_weight     * peak_hour_flag
  + valley_hour_weight   * valley_hour_flag
```

当前默认值：

- `transform = raw`
- `price_spread_scale_yuan_per_mwh = 100.0`
- `min_load_scale_mwh = 1.0`
- `signal_clip_abs = 0.0`
- `spread_weight = 0.02`
- `load_dev_weight = 0.01`
- `renewable_weight = 0.01`
- `spread_abs_weight = 0.005`
- `renewable_abs_weight = 0.004`
- `business_hour = 0.50`
- `peak_hour = 0.30`
- `valley_hour = -0.20`

小时级原始限制：

```text
raw_spot_hedge_limit_mwh
  = exposure_band_mwh / valid_hour_count
    * (base_multiplier + shrink_multiplier * policy_shrink)
```

当前代码这里使用制度投影后的 `exposure_band_mwh` 生成原始小时限制，随后在 `project_hourly_hedge_tensor()` 中继续按小时份额和爬坡约束进行正式投影。这样即使上层原始带宽被压到 0，可行域 floor 仍能为小时级修正保留合法交易空间。

小时门控：

```text
if gate_mode == "hard":
    if abs(hourly_signal) >= signal_deadband:
        gate = sigmoid((abs(hourly_signal) - deadband) / temperature)
    else:
        gate = 0

if gate_mode == "soft":
    gate = sigmoid((abs(hourly_signal) - deadband) / temperature)
```

当前默认：

- `hourly_gate.enabled = true`
- `hourly_gate.mode = hard`
- `signal_deadband = 0.12`
- `temperature = 0.04`
- `hourly_limit.base_multiplier = 0.50`
- `hourly_limit.shrink_multiplier = 0.50`

原始现货修正量：

```text
raw_spot_hedge_mwh
  = gate
    * tanh(hourly_signal)
    * raw_spot_hedge_limit_mwh
    * hour_mask
```

本轮保存粒子扫描同时比较 `hard` 和 `soft` 两种门控形式。`soft` 门控能保留死区内的小幅信号响应，部分候选通过 guardrail；完整远程重训显示扫描候选存在外推不稳，当前正式入口保留 `hard` 门控、`signal_deadband = 0.12` 和 `hourly_limit.shrink_multiplier = 0.50`。

投影后现货修正量：

```text
spot_hedge_mwh
  = Proj_hour(
      raw_spot_hedge_mwh,
      projected_exposure_band_mwh = exposure_band_mwh,
      hourly_share_cap,
      hourly_ramp_share
    )
```

`spot_hedge_mwh` 是带符号的小时级净修正量。正值代表增加现货侧计划电量，负值代表减少现货侧计划电量。摩擦成本按绝对成交量计算。

当前 `outputs/v0.51/raw/metrics/rolling_hourly_results.csv` 已由远程 Jupyter 全流程刷新。`spot_hedge_mwh` 共有 1632 条记录，非零小时数为 1535，带符号合计为 `-62549.650085` MWh，绝对成交量合计为 `416306.317581` MWh。

## 9. 15 分钟代理结算

评分核使用 `quarter_price_tensor` 在 15 分钟层计算成本、风险和收益。

### 9.1 周度底仓到 15 分钟

评分核先把中长期底仓按 24 小时曲线映射到小时，再映射到每个 15 分钟点：

```text
contract_interval_t = contract_hourly_h / intervals_per_hour_h
spot_interval_t     = spot_hedge_mwh_h / intervals_per_hour_h

scheduled_interval_t
  = max(contract_interval_t + spot_interval_t, 0)
```

若 15 分钟实际净负荷列可用，则直接使用 `net_load_id_mwh`；若不可用，则把周度实际净负荷均分到有效 15 分钟点。

### 9.2 成本口径

当前评分核的结算成本公式：

```text
interval_cost_t
  = scheduled_interval_t
    * (lt_settlement_weight * lt_price_w_effective
       + da_settlement_weight * da_price_t)
  + abs(actual_interval_t - scheduled_interval_t)
    * id_price_t
    * imbalance_penalty_multiplier
```

当前默认：

- `lt_settlement_weight = 0.60`
- `da_settlement_weight = 0.40`
- `imbalance_penalty_multiplier = 1.00`

周度采购成本：

```text
procurement_cost_w = sum(interval_cost_t)
```

售电收入：

```text
retail_revenue_w = actual_load_settlement_w * retail_tariff_yuan_per_mwh
```

当前 `retail_tariff_yuan_per_mwh = 430.0`。

利润：

```text
profit_w
  = retail_revenue_w
  - procurement_cost_w
  - adjustment_cost_w
  - friction_cost_w
```

其中：

```text
adjustment_cost_w
  = abs(contract_adjustment_mwh_raw - contract_adjustment_mwh_exec)
    * adjustment_cost_yuan_per_mwh

friction_cost_w
  = sum(abs(spot_hedge_mwh_h))
    * friction_cost_yuan_per_mwh
```

当前：

- `adjustment_cost_yuan_per_mwh = 0.60`
- `friction_cost_yuan_per_mwh = 1.50`

### 9.3 CVaR99

评分核按周对 15 分钟成本取右尾均值：

```text
tail_count = ceil(valid_intervals * (1 - cvar_alpha))
cvar99_w   = mean(top interval_cost_t over tail_count)
```

当前 `cvar_alpha = 0.99`。

### 9.4 强基准与超额收益

训练奖励中的强基准由 `reward.baseline_position_ratios` 控制，当前为：

```text
[0.50, 0.55, 0.60]
```

评分核对这些基准比例分别构造均匀中长期底仓，并计算采购成本。然后取采购成本最小的一组作为训练时的基准成本：

```text
baseline_procurement_cost_w
  = min(cost_w(ratio=0.50), cost_w(ratio=0.55), cost_w(ratio=0.60))

profit_baseline_w
  = retail_revenue_w - baseline_procurement_cost_w

excess_profit_w
  = profit_w - profit_baseline_w
```

该训练强基准与 `src/analysis/benchmarks.py` 中单独报告的 `dynamic_lock_only` 基准互相关联，但实现口径并非完全同一个函数。训练评分核使用比例族的最优采购成本，基准比较表中 `dynamic_lock_only` 固定为 `contract_ratio=0.55`。

### 9.5 奖励函数

周度奖励：

```text
reward_w
  = excess_profit_w
  - lambda_tail   * cvar99_w
  - lambda_hedge  * hedge_error_w
  - lambda_trade  * friction_cost_w
  - lambda_violate * policy_violation_penalty_w
```

当前默认权重：

- `lambda_tail = 0.65`
- `lambda_hedge = 0.18`
- `lambda_trade = 0.10`
- `lambda_violate = 1.00`

套保误差：

```text
hedge_error_w = imbalance_energy_w / actual_load_settlement_w
```

可行域惩罚：

```text
feasible_domain_clip_gap_w
  = abs(contract_adjustment_mwh_raw - contract_adjustment_mwh_exec)
  + abs(exposure_band_mwh_raw - exposure_band_mwh)
  + hourly_projection_gap_w

policy_violation_penalty_w
  = feasible_domain_clip_gap_w * violation_penalty_scale
```

总目标值：

```text
total_score = -sum(reward_w)
```

因此训练器最小化 `total_score`，等价于最大化全训练周奖励。

## 13. 当前实现边界与注意事项

### 13.1 价格联动标签与评分核价格

当前 `weekly_metadata` 在 2026-02 起写入 `lt_price_source=linked_mix_40da_60id`，并记录 `lt_price_fixed_ratio=0.4`、`lt_price_linked_ratio=0.6`。`prepare_project_context()` 同步生成 `lt_price_w_prev_week_da_proxy` 和 `lt_price_w_effective`，并把 `lt_price_w` 覆盖为 effective price。

`src/training/tensor_bundle.py` 优先读取 `lt_price_w_effective` 编译 `lt_weekly_price`，评分核、物化回测和基准策略均通过该张量取得同一价格口径。正式评分核按：

```text
lt_settlement_weight * lt_price_w_effective + da_settlement_weight * da_price_t
```

计算计划电量成本。`src/backtest/settlement.py::settle_week()` 仅作为参考实现保留，不属于当前全流程主路径。

### 13.2 小时级现货修正

小时级信号、门控、带宽、投影和摩擦成本在代码中完整存在。当前代码已将小时原始修正限额绑定到制度投影后的 `exposure_band_mwh`，远程刷新后的正式滚动输出已经产生非零现货修正。当前参数组合下仍应继续关注强基准族比较和尾部风险：滚动窗口跑赢强基准族的窗口数为 6，满足持续超额收益口径的窗口数为 0，持出集利润最高基准仍为 `static_no_spot_adjustment`。保存粒子扫描中的信号归一化、软门控、信号裁剪和更低死区候选没有在完整远程重训中形成更优外推结果，因此只作为后续研究变量构造与算法形式的候选空间保留。

### 13.3 配置维度与真实维度

`experiment_config.yaml` 当前保留 `hybrid_pso.upper.dimension: 182`，与当前 `compiled_parameter_layout` 的上层真实维度一致。训练器在传入 `compiled_parameter_layout` 时仍以真实编译维度为准，后续维护时报告和发布说明应优先引用真实编译维度。

### 13.4 远程与本地正式产物

远程 Jupyter 验证产物保存在 `outputs/v0.51/raw/remote_jupyter/` 下。当前 manifest 以本次最终产物为准写入，不再保留旧式分散报告索引。需要审计正式数值时，应优先比对 `raw/remote_jupyter` 回传快照、`run_manifest.json` 时间戳和当前本地产物。
