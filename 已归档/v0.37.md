# v0.37 Design

**Date:** 2026-04-16

**Scope:** 在不改变“双层参数化 HPSO + 小时级现货修正 + 15 分钟代理结算回测”正式主线的前提下，将 `v0.36` 的“参数真实进入模型”升级为“策略在制度可行域内生成、按阶段执行、可被关键事件解释”的现实化主线。`v0.37` 的核心工作不是继续扩维或增加搜索强度，而是把结算执行链、政策约束链、政策语义链、摩擦状态链和事件评估链真正打通。

---

## 1. 背景与目标

`v0.36` 已解决 `experiment_config.yaml -> 参数编译器 -> 训练核 -> 评分核` 的主链打通问题，使上下层特征与参数块能够真实进入 HPSO 搜索。但当前正式主线仍存在以下结构性不足：

1. 代理结算仍偏结果型映射，缺少“周度底仓 -> 日前申报 -> 小时级修正 -> 偏差暴露 -> 15 分钟结算 -> 周度汇总”的可追踪执行链。
2. 政策规则虽然已进入规则表、状态轨迹和报告，但多数情况下仍以后验惩罚或缩放方式作用于目标函数，尚未成为动作生成的前置可行域。
3. 政策文本到规则候选、制度状态、动作边界和结算口径之间的语义传递链条仍不够稳定，制度变化尚难形成可复核的因果解释。
4. 当前状态特征对交易摩擦、执行摩擦和制度摩擦的表达不足，难以解释真实经营中“为何不做某个看似更赚钱的动作”。
5. 评估层仍以均值、波动率、CVaR 和基准对比为主，尚不足以回答“关键事件发生时模型为什么这样响应、后果是什么”。

`v0.37` 的目标是完成以下四项正式升级：

1. 将代理结算改造为可执行、可追踪、可审计的阶段化执行链。
2. 将政策从目标函数附加惩罚前移为动作生成与结算选择的可行域生成器。
3. 将政策文本、结构化规则、制度状态、动作约束和结算口径串成稳定的语义编译链。
4. 将评估层升级为事件驱动解释框架，使模型能围绕关键市场事件输出“策略响应 -> 约束触发 -> 成本传导 -> 收益风险结果”。

---

## 2. 不变边界

以下边界在 `v0.37` 继续保持不变：

- 正式主算法仍为双层参数化 HPSO 主线，建议升级算法标签为 `HYBRID_PSO_V037`，但不恢复 PPO / SB3 / gym 或其他上层 DRL。
- 上层仍为周度中长期持仓参数化策略，下层仍为小时级现货滚动修正规则，不新增第三层市场优化。
- 正式结算与回测仍以 15 分钟代理结算为准，不单独训练实时市场策略。
- 湖南制度时点口径继续严格区分发布时间与生效时间，后期政策不得前移覆盖前期环境。
- 市场范围仍限定为中长期市场、日前现货市场、日内/实时现货市场。
- 零售端约束仍视为售电公司收益端约束，不转化为自由可调动作。
- `experiment_config.yaml` 仍为唯一人工修改入口，`run_all.py` 仍为人工一键入口。
- 正式产物仍写入 `outputs/<version>/<真实输出>`。
- 不引入 LLM 政策评分系统；`src/policy_deep/llm_bridge.py` 继续保持禁用，不进入正式主链。

`v0.37` 的定位是：在既有主算法和配置驱动主链不变的前提下，增强制度可执行性、执行可追踪性和结果可解释性。

---

## 3. 总体架构

`v0.37` 采用“政策语义编译 + 可行域生成 + 阶段化执行结算 + 事件驱动评估”的主链重构方案。

新的正式链路如下：

`政策文件/结果表`
-> `policy_parser / policy_tables`
-> `policy_semantic_compiler`
-> `policy_state_trace + feasible_domain_manifest`
-> `parameter_compiler + score_kernel + rolling_hedge`
-> `executable_settlement_engine`
-> `execution_trace / materialized_results`
-> `event_driven_evaluation / reporting`

### 3.1 四个正式子系统

1. `可执行代理结算主链`
   - 把现有偏结果型的代理结算改造成阶段化执行过程。
   - 输出中间执行结果，而不只输出最终成本和收益指标。

2. `政策可行域与语义编译链`
   - 将政策文本中的关键制度信息稳定映射为结构化规则、制度状态、动作边界、参数块启停和结算口径切换。
   - 让策略在生成阶段就处于制度可行域之内。

3. `摩擦状态扩展`
   - 将交易摩擦、执行摩擦、制度摩擦正式纳入周度/小时级状态空间。
   - 让模型更真实表达“收益、合规、执行成本”之间的权衡。

4. `事件型评估与报告`
   - 以关键事件为中心组织评估，而不是只做平均效果比较。
   - 保持 `dynamic_lock_only` 为强基准，重点解释主动管理策略在事件期为何有效或失效。

### 3.2 设计原则

- 先建立执行链，再前移政策可行域，再补语义链，再扩摩擦状态，最后做事件解释。
- 新增复杂度必须服务于制度一致性与结果解释，不以拉长计算时间为目的。
- 所有新增字段都必须说明来源、时点标签、是否进入 agent、是否进入结算、是否仅用于报告。
- 所有新增链路都必须避免未来信息泄漏，特别是事件识别与制度切换标签。

## 4. 可执行代理结算主链设计

### 4.1 问题定义

当前 `src/backtest/settlement.py` 更接近“给定动作 -> 汇总型结算结果”的快速映射，虽然可满足训练和回测，但不足以支撑以下需求：

- 解释周度底仓如何通过日前和日内执行路径影响最终结算。
- 判断策略收益提升是否来自合理执行，而非某种压缩后的口径偏差。
- 为事件评估提供阶段化中间变量，而不是只比较周度总成本。

### 4.2 正式阶段链

`v0.37` 中，正式执行链至少显式经过以下阶段：

1. `周度底仓形成`
   - 输入：`lock_ratio_base`、`delta_lock_ratio`、`lock_ratio_final`、制度状态、周度负荷预测。
   - 输出：`weekly_contract_position_mwh`、`weekly_contract_curve_24h`、`weekly_exposure_band_mwh`。

2. `日前申报展开`
   - 将周度底仓和 24 点合约曲线映射到运行日级别。
   - 输出：`day_ahead_contract_schedule`、`day_ahead_exposure_reference`、`declaration_mode`。

3. `小时级现货滚动修正`
   - 在既定底仓基础上，根据价差、负荷偏差、新能源扰动和摩擦变量进行滚动修正。
   - 输出：`hourly_delta_q_target`、`hourly_delta_q_executed`、`execution_clamp_reason`。

4. `偏差暴露与代理持仓形成`
   - 将日前/日内执行结果与真实负荷、代理现货价格轨迹组合，形成小时级偏差暴露。
   - 输出：`hourly_residual_exposure_mwh`、`hourly_imbalance_cost_basis`。

5. `15 分钟代理结算`
   - 将小时级执行结果展开或映射到 96 点结算粒度。
   - 输出：`quarter_hour_settlement_trace`，至少包括结算电量、价格、偏差、电费和制度口径标签。

6. `辅助服务与相关分摊影响`
   - 将制度相关分摊、紧约束状态和代理辅助服务影响纳入结算后处理。
   - 输出：`ancillary_allocation_cost`、`policy_allocation_note`。

7. `周度汇总与策略结果物化`
   - 将阶段结果汇总为训练、验证、回测和报告层统一消费的 `weekly_results`。

### 4.3 输出原则

执行链必须支持三层输出：

1. `stage summary`
   - 面向训练/回测主线的轻量摘要。

2. `hourly execution trace`
   - 面向诊断和事件评估的小时级执行轨迹。

3. `quarter-hour settlement trace`
   - 面向最终制度结算解释的 15 分钟轨迹。

默认运行不必保存全部明细，但验证和回测阶段必须能在配置开启时稳定导出。

### 4.4 代码落点

`v0.37` 建议按以下边界实施：

- 修改：`src/backtest/settlement.py`
  - 从汇总型结算函数扩展为阶段化执行结算主入口。
- 新增：`src/backtest/execution_trace.py`
  - 负责阶段性中间变量组织、标准列名和 trace 汇总。
- 修改：`src/backtest/materialize.py`
  - 将执行链中间结果稳定写入 `weekly_results` 与诊断产物。
- 修改：`src/rules/rolling_hedge.py`
  - 输出真正可解释的执行前目标值、执行后值和收缩原因。
- 修改：`src/backtest/rolling_pipeline.py`
  - 在滚动验证和回测中传递 `execution_trace` 与阶段摘要。

## 5. 政策可行域与语义编译链设计

### 5.1 目标

`v0.37` 中，政策不再主要作为“违规后扣分”的附加惩罚，而应在动作生成前就参与以下环节：

- 动作边界生成
- 参数块启停
- 结算口径切换
- 报告解释标签生成

这意味着正式主链需要从“policy projection / penalty”升级为“policy feasible domain”。

### 5.2 正式语义链

正式语义链定义为：

`政策文本片段`
-> `规则候选`
-> `结构化规则表`
-> `当前制度状态 + 四组前瞻制度状态`
-> `可行域清单`
-> `上层动作边界 / 下层收缩规则 / 结算口径选择`

其中：

- `政策文本片段 -> 规则候选`
  - 继续基于 `src/policy/policy_parser.py`、`src/policy_deep/document_reader.py`、`src/policy_deep/candidate_rules.py` 的确定性解析实现。
- `规则候选 -> 结构化规则表`
  - 继续落地为项目要求中的 `rule_id`、`policy_name`、`publish_time`、`effective_start`、`effective_end`、`rule_type`、`scope`、`state_group`、`state_name`、`state_value`、`source_file`、`note`。
- `结构化规则表 -> 制度状态`
  - 继续由 `src/policy/policy_regime.py` 生成当前制度状态和四组前瞻制度状态。
- `制度状态 -> 可行域清单`
  - 新增正式编译层，输出每周/每日/小时的动作边界、参数块启停和结算口径切换条件。

### 5.3 新增正式组件

建议新增两个明确模块：

- 新增：`src/policy_deep/semantic_compiler.py`
  - 职责：将规则候选和规则表编译为可行域相关语义标签。
  - 输出：`policy_semantic_mapping`、`state_binding_summary`。

- 新增：`src/policy/feasible_domain.py`
  - 职责：将当前制度状态和四组前瞻状态映射为动作边界、参数块启停、结算模式切换和结算约束标签。
  - 输出：`feasible_domain_manifest`、`action_bound_trace`、`settlement_mode_trace`。

### 5.4 对训练核和规则核的影响

- `src/training/score_kernel.py`
  - 不再只在奖励层吸收政策惩罚，而要消费 `feasible_domain_manifest`，在动作构造和下层修正规则中显式使用可行域边界。
- `src/agents/hybrid_pso.py`
  - runtime profile 需要记录本轮训练实际使用的可行域版本、绑定制度状态组和关键边界摘要。
- `src/scripts/common.py`
  - 在 `prepare_project_context()` 中统一完成政策解析、语义编译、可行域构建，并将结果传递到训练、验证和回测全链路。

### 5.5 配置口径

建议将当前 `policy_projection` 升级为更正式的 `policy_feasible_domain` 节，并保留一版兼容映射。

建议最少包含：

- `enabled`
- `strict_mode`
- `bind_upper_actions`
- `bind_lower_actions`
- `bind_settlement_mode`
- `mandatory_state_groups`
- `non_negative_position_required`
- `forward_state_horizon_groups`
- `export_action_bound_trace`

### 5.6 产物要求

在保留现有政策清单、元数据索引、规则表和解析失败清单的基础上，`v0.37` 额外新增：

- `outputs/<version>/metrics/policy_semantic_mapping.csv`
- `outputs/<version>/metrics/feasible_domain_manifest.csv`
- `outputs/<version>/reports/policy_feasible_domain_summary.md`

这些产物必须明确说明：

- 哪些制度状态绑定了哪些动作边界
- 哪些参数块在何种制度下被启停或收缩
- 哪些结算口径切换由何项规则触发

## 6. 摩擦状态扩展设计

### 6.1 设计目标

新增状态不是为了机械增加维度，而是为了把“真实经营中阻止某些动作发生的摩擦”正式纳入模型。每个新增摩擦变量都必须回答三个问题：

1. 它约束什么动作或执行过程？
2. 它由哪些现有数据和制度状态生成？
3. 它将在报告中如何被解释？

### 6.2 三类正式摩擦变量

#### 交易摩擦

- 成交窗口限制
- 曲线调整剩余空间
- 成交平滑压力
- 流动性收缩代理

#### 执行摩擦

- 负荷偏差压力
- 现货修正切换成本
- 对冲连续性压力
- 代理偏差考核压力

#### 制度摩擦

- 制度切换倒计时压力
- 当前绑定规则数量
- 价格机制锁定状态
- 辅助服务耦合收紧状态

### 6.3 数据落点

建议优先从现有周度/小时级字段和制度状态中派生，不新增论文主线外原始数据源。实现上建议：

- 修改：`src/data/weekly_builder.py`
  - 补充周度摩擦变量的统一派生。
- 修改：`src/data/feature_engineering.py`
  - 补充小时级摩擦特征或中间派生逻辑。
- 修改：`src/model_layout/compiler.py`
  - 支持将摩擦变量按显式参数块纳入上层/下层编译布局。
- 修改：`src/training/tensor_bundle.py`
  - 将新增摩擦状态稳定打包到训练核输入张量。

### 6.4 参数布局原则

`v0.37` 不应把摩擦变量直接散落到既有 `weekly_feature_weights` 中而失去解释性。建议在 `parameter_compiler` 中显式新增摩擦相关参数块，例如：

- 上层：`friction_feature_weights`
- 下层：`execution_friction_response`

这样可以保持“常规市场状态”和“摩擦状态”在参数布局和报告解释上的可区分性。

## 7. 事件型评估与报告设计

### 7.1 评估升级方向

`v0.37` 评估层从“总体均值优于基准”升级为“关键事件中响应是否合理且结果是否优于基准”。正式评估对象包括：

- 价格突变事件
- 政策切换事件
- 负荷偏差事件
- 新能源扰动事件
- 辅助服务收紧事件

### 7.2 正式事件链

`事件识别`
-> `事件窗口截取`
-> `策略响应摘要`
-> `制度约束触发摘要`
-> `执行链成本传导摘要`
-> `相对 dynamic_lock_only 的结果比较`

### 7.3 新增正式模块

- 新增：`src/analysis/event_evaluation.py`
  - 负责事件识别、窗口切片、响应摘要、基准比较和结论输出。
- 修改：`src/analysis/reporting.py`
  - 将事件评估结果整合到正式报告中。
- 修改：`src/scripts/evaluate.py`
  - 在验证期生成事件评估正式产物。
- 修改：`src/scripts/backtest.py`
  - 在测试期和滚动窗口生成事件评估正式产物。

### 7.4 正式产物

建议新增：

- `outputs/<version>/metrics/event_catalog.csv`
- `outputs/<version>/metrics/event_response_summary.csv`
- `outputs/<version>/reports/event_driven_evaluation.md`

其中 `event_response_summary.csv` 每个事件至少包含：

- `event_id`
- `event_type`
- `event_start`
- `event_end`
- `policy_state_snapshot`
- `strategy_action_change`
- `feasible_domain_triggered`
- `execution_cost_pass_through`
- `excess_profit_vs_dynamic_lock_only`
- `tail_risk_delta`
- `event_conclusion`

### 7.5 与既有评估指标的关系

- 保留累计收益、均值、波动率、最大回撤、尾部风险指标和基准比较。
- 事件评估是对总体评估的补强，而不是替代。
- 所有事件结论都必须可回溯到执行链与制度状态，而不是脱离轨迹的事后描述。

## 8. 入口集成设计

### 8.1 `prepare_project_context`

`src/scripts/common.py` 中的 `prepare_project_context()` 需要成为 `v0.37` 的正式预编译入口，统一完成：

1. 政策文件扫描、元数据索引、规则表和失败清单生成。
2. 语义编译与可行域清单生成。
3. 参数编译器对摩擦变量与新参数块的编译。
4. 将 `policy_state_trace`、`feasible_domain_manifest`、`compiled_layout` 一并注入上下文。

### 8.2 `train`

`src/scripts/train.py` 继续驱动 HPSO 训练，但需确保：

- 训练核读取的是可行域约束后的动作生成链。
- runtime summary 记录语义编译版本、可行域摘要和执行链版本。

### 8.3 `evaluate`

`src/scripts/evaluate.py` 在验证物化后额外执行：

- 执行链 trace 导出
- 政策可行域摘要导出
- 事件评估导出

### 8.4 `backtest`

`src/scripts/backtest.py` 在滚动回测中额外执行：

- 窗口级执行链物化
- 测试周事件评估
- 关键事件基准对比摘要

### 8.5 `run_pipeline` 与 `run_all.py`

- `src/scripts/run_pipeline.py` 仍保持训练 -> 验证 -> 回测主顺序。
- `run_all.py` 仍无需新增人工切换入口，但运行摘要应引用 `v0.37` 新增报告和 trace 产物。

## 14. 风险与控制

主要风险包括：

1. `政策语义 -> 可行域 -> 结算口径` 口径不一致，导致制度解释失真。
2. 执行 trace 粒度过细，导致验证和回测 I/O 过重。
3. 事件识别错误使用未来信息，导致评估泄漏。
4. 为兼容旧代码而保留过多后验惩罚路径，导致“前置可行域”目标落空。
5. 摩擦变量名义进入配置，但未真正进入训练核和评分核。

控制方式：

- 用统一的 `semantic_compiler` 和 `feasible_domain` 输出结构，避免多处手工推断。
- 将 trace 输出分为默认轻量摘要和按需详细导出两层。
- 事件识别严格基于事件发生时点可观测信息构造。
- 明确将旧的纯惩罚式政策路径降级为兼容回退，不再作为正式主逻辑。
- 在 `train_summary.md`、`validation_summary.md` 和参数布局摘要中同时记录摩擦参数块与可行域摘要，确保可审计。

## 15. 不在 `v0.37` 范围内的内容

以下内容不纳入 `v0.37`：

- 恢复 PPO / SAC / 独立下层 DRL 或其他新训练范式
- 引入 LLM 政策评分系统或在线政策解释主链
- 引入第三层市场优化、辅助服务收益优化或论文未设定的新市场主体
- 继续单纯扩大粒子规模或参数维度而缺乏制度与执行含义
- 以展示型图表替代执行链、制度链和事件链的正式产物

`v0.37` 的正式定位是：把 `v0.36` 已打通的参数化双层 HPSO 主线，进一步升级为“制度前置、执行可追踪、结果可解释”的现实化正式版本。
