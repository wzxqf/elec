# elec 项目 v0.4 修改方案（真实约束收口版 + 论文可证性增强版）

## 一、文档目的

本文档用于明确 `elec` 项目后续 `v0.4` 版本的改造目标、问题诊断、模块级修改内容、实施顺序、测试验收要求与论文写作边界。其定位不是继续外扩算法种类，而是在现有 `v0.36` 结果基础上，完成版本口径统一、训练与回测链路收口、政策约束语义做实、指标体系稳定化与复现性增强，使项目从“可运行、可出结果的策略原型”进一步推进为“口径统一、约束可解释、结果可复核的论文实验系统”。

---

## 二、现状判断与主要问题

### 2.1 当前项目已经具备的基础

从现有仓库结构和输出结果看，项目已经具备较完整的实验闭环，包括：

- 数据清洗、频率校验与质量报告；
- 政策文件清单、规则结构化与制度状态轨迹；
- 参数编译、训练、验证、滚动回测与报告输出；
- 强基准 `dynamic_lock_only` 对比；
- 训练摘要、回测摘要、数据质量报告等实验产物。

这意味着项目已经可以支撑论文中的研究背景、模型设计、实验实现与结果展示四个基本部分。

### 2.2 当前版本最核心的不足

#### （1）版本口径分裂

当前仓库首页 README 仍以 `v0.31` 作为维护口径，而架构文档已经写到 `v0.38`，训练产物则输出到 `v0.36`。这会导致方法、实现、结果三部分的版本边界不一致，论文写作时极易出现“文档写的是一个系统，实验跑的是另一个系统”的问题。

#### （2）政策约束的正式语义尚未真正做实

目前正式文档已经把主语义定义为“政策投影约束”，并明确排除了额外极端裁剪链。但从现有评分核结构看，政策作用仍主要表现为对业务动作进行连续收缩，并通过惩罚项反映偏离程度。换言之，当前实现更接近“缩放 + 惩罚”的代理约束，而不是“政策状态生成可行域，再将动作投影到可行域内部”的可执行约束。

#### （3）状态变量的现实厚度仍偏薄

当前训练张量中，小时级特征仍较为精简，15 分钟层也主要保留价格列，尚未形成“制度状态—价格结构—负荷偏差—极端事件”之间的多尺度状态表达。这会削弱模型对现实波动、异常周和制度状态切换的响应能力。

#### （4）指标体系中存在不稳定项

当前回测摘要中已经出现极端的风险调整指标值，这说明部分比率类指标在窗口较短、分母过小或尺度不稳定时仍可能失真。若不处理，论文中很难直接引用这些指标并作出稳健解释。

#### （5）样本长度和稳健性证据仍有限

现阶段主样本期覆盖时间仍偏短，滚动窗口数量也有限。对于原型系统，这已经足够产出初步结论；但若要在论文中形成较强的外推性表述，现有样本和稳健性设计仍然不足。

#### （6）复现性工程仍不完整

当前报告中仍保留绝对路径、设备依赖和版本边界不统一等问题。项目虽已有较强的可追踪性，但尚未形成面向论文附录、答辩复核和多机复现实验的统一 artifact 清单。

## 三、v0.4 的版本定位

### 3.1 总体定位

`v0.4` 的定位不是“继续增加算法复杂度”，而是：

> 在既有 HPSO/参数化策略框架不大改的前提下，完成真实约束收口、训练与回测口径统一、状态变量增强、指标稳定化与实验复现规范化。

### 3.2 版本目标

`v0.4` 需要同时实现以下五项目标：

1. **统一版本口径**：方法、文档、配置、输出和报告使用同一版本号。
2. **打通主训练链路**：参数编译布局必须在训练、验证、滚动回测、物化与报告中一致传递。
3. **做实政策约束**：将政策从惩罚项主导的代理约束，推进为显式可行域生成与投影约束。
4. **增强现实状态表达**：补入真正驱动购电决策的多尺度状态变量。
5. **提高论文可证性**：建立更稳健的指标体系、基准矩阵、消融设计与复现材料。

### 3.3 非目标

为避免版本边界失控，`v0.4` 暂不追求以下事项：

- 不新增全新的主优化算法；
- 不把系统重构为完全不同的 DRL 框架；
- 不在本版本中追求“直接部署级”生产系统；
- 不在尚未收口约束语义前继续外扩更多复杂功能。

---

## 四、总体改造思路

`v0.4` 应围绕以下主线进行重构：

```text
data
  -> policy_deep
  -> feasible_domain / policy_projection
  -> tensor_bundle
  -> parameter_compiler
  -> parametric_hybrid_pso
  -> rolling_retrain
  -> 15m_settlement
  -> reporting / audit
```

与现阶段相比，`v0.4` 最重要的变化不是再增加一层优化器，而是把 `policy_projection` 从文档语义推进为真实执行模块，并将其前移到训练热路径中，保证“政策状态 -> 可行域 -> 动作投影 -> 结算结果”形成可审计链路。

## 五、模块级修改方案

## 5.1 版本治理与文档收口

### 修改目标

彻底解决 README、架构文档、约束文档、输出目录与实验报告之间的版本分裂问题。

### 具体修改

1. 将项目正式版本统一命名为 `v0.4`。
2. 同步修改以下文件中的版本号与版本表述：
   - `README.md`
   - `CHANGELOG.md`
   - `docs/ARCHITECTURE.md`
   - `docs/CONSTRAINTS.md`
   - `experiment_config.yaml`
   - `outputs/` 中新版本目录
3. 新增 `outputs/v0.4/release_manifest.json`，记录：
   - 版本号
   - 配置文件路径与 hash
   - 样本时间范围
   - 训练周与验证周
   - 设备信息
   - 随机种子
   - 参数编译布局路径
   - 启用的约束清单
   - 关键输出文件路径
4. 所有报告页眉统一输出：
   - version
   - experiment_id
   - config_hash
   - run_timestamp
   - device
   - data_range

### 预期效果

版本号、实现边界和结果产物一一对应，论文引用与代码实现可以互相对照，避免口径漂移。

## 5.2 参数编译布局全链路收口

### 修改目标

确保 `parameter_compiler -> compiled_layout -> score_kernel/materialize/backtest/reporting` 构成唯一有效主链路，避免训练、验证、回测之间存在隐式退化分支。

### 具体修改

1. 复核并强制以下入口显式接收 `compiled_layout`：
   - 训练入口
   - 验证入口
   - 滚动回测入口
   - materialize 入口
   - 结果审计入口
2. 在 `compiled_layout is not None` 时，禁止进入旧的“按最小维度截断”或“隐式默认维度”路径。
3. 在运行摘要中新增：
   - `upper_dim_real`
   - `lower_dim_real`
   - `compiled_layout_path`
   - `compiled_layout_hash`
4. 增加一致性校验：
   - 同一组粒子在训练态、验证态、回测态上的打分结果应在容忍误差内一致；
   - 若出现维度缺失、块缺失或布局不匹配，直接报错而非静默回退。
5. 增设审计报告：
   - `parameter_layout_audit.md`
   - 列出每一参数块的起止索引、业务含义、是否生效、在评分核中的消费位置。

### 预期效果

论文可以明确写成“参数编译布局已进入主训练热路径，并在训练—验证—回测中保持一致”。

---

## 5.3 政策约束从惩罚项推进为可行域生成器

### 修改目标

将现有“缩放 + 惩罚”的代理约束机制，升级为“政策状态生成可行域 -> 动作投影 -> 诊断输出”的正式约束机制。

### 新增模块

新增：

- `src/policy/feasible_domain.py`
- `src/policy/projection.py`

### 职责划分

#### `feasible_domain.py`

根据当周制度状态、规则标识和市场条件，生成业务动作的可行域对象，包括但不限于：

- 周度合约调整上下界；
- 敞口允许带宽；
- 调整斜率或变动速率边界；
- 曲线形状约束；
- 特殊制度阶段的额外收紧规则；
- 允许或禁止的修正类型。

#### `projection.py`

对上层和下层动作进行正式投影，输出：

- `raw_action`
- `projected_action`
- `projection_gap`
- `projection_reason_codes`

### 具体修改

1. `score_kernel.py` 不再以 `policy_scale` 作为主要正式业务边界。
2. `policy_scale` 若继续保留，只能作为诊断信息，不再承担正式业务约束语义。
3. 所有正式业务动作在进入成本、收益、结算环节前，必须先经过 `project_action_to_domain()`。
4. 生成约束审计表：
   - 当周可行域边界
   - 被触发的投影规则
   - 动作被压缩的幅度
   - 投影前后差异
5. 在报告中增加“约束激活痕迹”章节，用于解释某些窗口表现波动的制度原因。

### 预期效果

论文中可将政策约束表述为“制度状态驱动的可行域约束”，而非简单的惩罚项近似。

## 5.4 训练张量与状态变量增强

### 修改目标

在不显著破坏现有训练效率的前提下，补入能够真实影响售电公司采购决策的多尺度状态变量。

### 当前存在的问题

当前小时级特征集较薄，15 分钟层主要保留价格列，尚不足以完整反映：

- 日前与日内价差结构；
- 负荷偏差与新能源偏差的方向性；
- 极端事件冲击；
- 制度状态切换；
- 相关性突变。

### 修改方向

#### （1）周度层

建议新增或正式并入以下周度状态：

- 日前—日内相关性指标；
- 极端价格事件标识；
- 极端负荷或波动周标识；
- 制度状态切换标识；
- 风险预警型 regime 特征。

#### （2）小时级

建议增强：

- 价差分解特征；
- 负荷预测误差分位信息；
- 新能源偏差方向与幅度；
- 峰谷时段位置信息；
- 关键业务小时掩码。

#### （3）15 分钟层

建议在现有价格张量基础上加入：

- 价格有效性标识；
- 特殊时段掩码；
- 结算有效区间指示；
- 必要的异常值保护字段。

### 具体修改

1. 修改 `weekly_builder.py`，生成扩展状态字段。
2. 修改 `tensor_bundle.py`，统一输出：
   - `weekly_feature_columns`
   - `policy_columns`
   - `hourly_feature_columns`
   - `quarter_feature_columns`
3. 新增字段映射文件：
   - `docs/STATE_SCHEMA.md`
4. 增加张量审计输出：
   - 实际进入训练的字段名、维度、缺失填补方式、标准化方式。

### 预期效果

模型可以更真实地响应异常周、极端价差周和制度状态切换周，论文中的状态空间设计也更有现实厚度。

## 5.5 评分核重构与语义分层

### 修改目标

将 `score_kernel.py` 从“业务动作生成 + 惩罚项结算混合体”，重构为语义更清晰的三层结构。

### 推荐分层

1. **动作生成层**
   - 基于参数块和状态变量生成原始动作。
2. **约束投影层**
   - 将原始动作投影到政策可行域。
3. **经济结算层**
   - 基于投影后动作计算采购成本、偏差成本、收益、CVaR 与超额收益。

### 具体修改

1. 明确区分以下字段：
   - `raw_contract_adjustment`
   - `projected_contract_adjustment`
   - `contract_position`
   - `exposure_band`
2. 将“政策惩罚”拆分为：
   - 正式投影差额诊断；
   - 数值稳定性保护；
   - 仅用于解释的附加风险提示。
3. 把“数值保护”和“业务约束”彻底分开，避免论文表述混淆。
4. 对每一类损益项单独输出中间结果：
   - 采购成本
   - 调整成本
   - 摩擦成本
   - 机会损失
   - 风险惩罚
   - 基准收益
   - 超额收益
5. 为中间字段建立固定命名规范，供报告与论文结果图直接调用。

### 预期效果

模型逻辑更清楚，结果解释更自然，代码也更适合做消融实验和论文图表输出。

---

## 5.6 指标体系稳定化

### 修改目标

修复风险调整指标极端失真问题，使结果可以直接进入论文正文而不引发解释困难。

### 具体修改

1. 对 Sharpe 类指标增加：
   - 分母下限保护；
   - 样本数阈值；
   - 近零波动警告；
   - 极值告警。
2. 将单一 Sharpe 指标扩展为更稳健的指标组：
   - 平均收益
   - 收益中位数
   - 收益分位数
   - 波动率
   - CVaR95 / CVaR99
   - 最大回撤
   - 持续跑赢窗口数
   - 胜率
3. 新增指标解释页：
   - 指标定义
   - 计算口径
   - 年化方式
   - 窗口尺度
   - 异常值触发条件
4. 对不稳定指标建立“仅附录展示”与“正文可用”的分级机制。

### 预期效果

论文不会再依赖单个极端比率作结论，整体解释会更稳妥。

---

## 5.7 样本、回测与稳健性设计增强

### 修改目标

提升实验对论文结论的支撑强度。

### 具体修改

1. 若数据可得，尽量扩展样本期，补足不同季节与不同制度阶段。
2. 若短期内无法扩样，则必须强化滚动设计：
   - 滚动再训练；
   - 扩展窗再训练；
   - 固定训练窗 + 滑动测试窗对照。
3. 补齐基准矩阵：
   - `dynamic_lock_only`
   - 简化版本 HPSO
   - 去掉政策投影的版本
   - 去掉扩展状态变量的版本
4. 必做消融实验：
   - 无可行域投影
   - 无新增状态变量
   - 无参数编译布局增强
5. 输出稳健性报告：
   - `robustness_summary.md`
   - `ablation_summary.md`

### 预期效果

论文可以回答“模型为什么有效”，而不仅仅是“模型跑出来更好”。

## 七、实施顺序与阶段安排

## 7.1 P0：必须先完成的收口项

### 目标

在不改变总体算法框架的前提下，先解决最影响论文可信度的问题。

### 任务

1. 统一版本号到 `v0.4`；
2. 收口 `compiled_layout` 全链路；
3. 稳定风险调整指标；
4. 统一报告页眉与 artifact 索引；
5. 修复绝对路径输出问题。

### 完成标准

- 版本口径一致；
- 训练/验证/回测不再存在隐式退化分支；
- 指标无明显失真极值；
- 正式实验输出可一键归档。

---

## 7.2 P1：约束语义做实

### 目标

把政策约束从文档表述推进为真实执行模块。

### 任务

1. 实现 `feasible_domain.py`；
2. 实现 `projection.py`；
3. 将投影接入评分核主链；
4. 输出约束激活报告；
5. 清理旧的业务裁剪语义。

### 完成标准

- 所有正式业务动作均先投影后结算；
- 约束边界可以被解释、审计和复核；
- 文档、代码与报告对政策约束的表述一致。

## 九、面向论文写作的建议表述边界

在 `v0.4` 完成前，论文不宜写成“已形成现实中可直接执行的交易决策系统”。更稳妥的写法是：

> 本文构建了一个面向湖南电力市场售电公司采购决策的多时间尺度实验系统，在制度约束映射、参数化策略搜索、滚动回测和结果审计等方面实现了较完整的实验闭环，并在可行域约束、状态变量增强和滚动稳健性验证基础上，对购电策略的收益—风险权衡进行了系统分析。

若 `v0.4` 中的可行域约束、布局收口与稳健性实验完成得较好，则可以进一步写成：

> 本文在既有策略优化框架基础上，将政策规则转化为显式可行域，并通过滚动再训练与多组消融实验验证了该约束机制与状态增强设计对策略表现的影响。

这种表述更符合本科论文的稳妥写法，也更接近当前项目真正能够支撑的结论强度。

## 十、风险提示

`v0.4` 的风险主要不在“实现难度”，而在“边界失控”。若在本版本中同时推进过多新算法、新目标函数和新状态定义，极易出现以下问题：

- 版本定位模糊；
- 指标变化无法归因；
- 文档滞后于实现；
- 论文难以清楚解释改进来源。

因此，`v0.4` 必须坚持“先收口，再增强”的原则。只有在版本语义、约束语义和实验口径都统一之后，后续版本才适合继续探索更大规模的策略外扩。

## 十二、结论

从项目演进阶段看，`v0.4` 不应再理解为单纯的“功能更新版”，而应理解为**论文主版本的收口版**。它承担的任务不是让系统看上去更复杂，而是让系统的约束表达、训练口径、结果指标与论文表述真正统一起来。只要这一点做实，项目就能从“有结果的原型”迈向“能支撑论文答辩与复核的实验系统”；反之，如果继续在版本分裂、约束代理化和指标失真状态下外扩功能，论文的可信度反而会下降。
